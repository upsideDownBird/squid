## Context

当前 jobopx-desktop 有 5 个基础工具（ReadFile, WriteFile, Grep, Glob, SaveMemory），但缺少网络访问能力。用户在开发过程中经常需要查阅在线文档、API 参考等，目前只能手动复制粘贴。

claude-code-main 实现了成熟的 WebFetchTool，具备：
- HTML 到 Markdown 转换
- AI 模型内容摘要
- URL 缓存机制
- 预批准域名支持

## Goals / Non-Goals

**Goals:**
- 实现基础的网页获取和内容提取功能
- 支持 HTML 到 Markdown 转换
- 实现简单的缓存机制
- 符合现有工具规范（映射方法、持久化）

**Non-Goals:**
- 不实现域名安全检查（简化版本）
- 不实现预批准域名列表（可后续添加）
- 不实现二进制内容处理（PDF 等）
- 不实现 AI 内容摘要（简化版本，直接返回 Markdown）

## Decisions

### 1. 简化实现策略
**决策**: 实现简化版 WebFetch，只保留核心功能

**理由**:
- claude-code-main 的实现包含大量企业级功能（域名检查、安全策略）
- 我们的项目是个人/小团队使用，不需要复杂的安全机制
- 先实现核心功能，后续按需扩展

**简化内容**:
- ❌ 域名安全检查（需要调用 Anthropic API）
- ❌ 预批准域名列表
- ❌ AI 内容摘要（需要额外 API 调用）
- ✅ 保留：HTTP 请求、HTML 转换、缓存

### 2. HTML 转换库选择
**决策**: 使用 `turndown` 库

**理由**:
- claude-code-main 使用的库，成熟稳定
- 轻量级，专注 HTML 到 Markdown 转换
- 支持自定义规则

**替代方案**:
- `html-to-md` - 功能较少
- 自己实现 - 工作量大，容易出错

### 3. HTTP 客户端选择
**决策**: 使用 `axios`

**理由**:
- 功能完整，支持拦截器、超时等
- 与 claude-code-main 保持一致
- 社区活跃，文档完善

**替代方案**:
- `fetch` API - 浏览器原生，但 Node.js 支持较新
- `node-fetch` - 需要额外依赖

### 4. 缓存策略
**决策**: 使用 `lru-cache`，15 分钟 TTL，50MB 限制

**理由**:
- 避免重复请求相同 URL
- LRU 策略自动淘汰旧数据
- 参考 claude-code-main 的经验值

### 5. 输出格式
**决策**: 直接返回 Markdown 内容，不调用 AI 摘要

**理由**:
- 简化实现，减少 API 调用成本
- 用户可以自己阅读完整内容
- 如果内容太大，持久化系统会自动处理

## Risks / Trade-offs

### 风险 1: 网页内容过大
**风险**: 某些网页转换后的 Markdown 可能非常大

**缓解措施**:
- 设置 `maxResultSizeChars: 100000`（100K）
- 超过阈值自动触发持久化系统
- 返回预览 + 文件路径

### 风险 2: 网络请求失败
**风险**: 目标网站可能不可访问、超时、返回错误

**缓解措施**:
- 设置合理的超时时间（30 秒）
- 捕获所有异常并返回友好错误消息
- 支持 AbortController 取消请求

### 风险 3: HTML 解析失败
**风险**: 某些网页结构复杂，转换可能失败或效果差

**缓解措施**:
- 使用成熟的 turndown 库
- 如果转换失败，返回原始 HTML
- 记录错误日志便于调试

### Trade-off: 功能 vs 复杂度
**权衡**: 简化版本功能较少，但实现简单、维护容易

**选择**: 接受功能限制，优先快速交付可用版本

## Migration Plan

### 实施步骤
1. 安装依赖（turndown, axios, lru-cache）
2. 实现 web-fetch-utils.ts（HTTP 请求、HTML 转换、缓存）
3. 实现 web-fetch.ts（工具主体）
4. 在 registry 中注册工具
5. 测试基本功能

### 测试计划
- 测试获取简单 HTML 页面
- 测试缓存机制
- 测试错误处理（404、超时等）
- 测试大内容持久化

### 回滚策略
- 新工具独立，不影响现有功能
- 如有问题，从 registry 移除即可
- 无数据迁移，回滚无风险

## Open Questions

   - 建议：暂不支持，后续可添加 headers 参数

2. **是否需要限制可访问的域名？**
   - 建议：暂不限制，信任用户判断

3. **缓存清理策略？**
   - 建议：使用 LRU 自动淘汰，无需手动清理
