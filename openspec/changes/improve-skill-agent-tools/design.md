## Context

当前 `skill` 与 `agent` 工具均为“可运行但简化”的实现：`skill` 主要返回技能文本摘要，`agent` 通过独立 SDK 直连模型。与此同时，系统已有 `TaskAPI` + `TaskExecutor` + `ToolRegistry` 的统一执行链路。两套路径并存会带来行为不一致（配置来源、错误格式、结果映射格式不同）、排障复杂和测试覆盖碎片化。

本次设计在不引入新依赖前提下，优先复用现有执行栈，并参考 `claude-code-main` 的思路：执行入口统一、能力可组合、结果可观测且可映射。

## Goals / Non-Goals

**Goals:**
- 让 `skill` 工具具备“执行语义”，而不是仅做静态展示。
- 让 `agent` 工具复用统一执行栈和配置来源，避免直连分叉。
- 统一 `skill` / `agent` 的成功失败语义、错误分类和输出映射。
- 为后续扩展（多模型、子代理策略、权限策略）保留稳定接口。

**Non-Goals:**
- 不在本次引入全新沙箱或进程级隔离系统。
- 不实现完整的多轮 agent 编排平台（仅完成当前工具能力升级）。
- 不修改已有非相关工具的接口契约。

## Decisions

### Decision 1: `agent` 工具改为通过统一执行栈执行

- 方案：`agent` 不再直接调用单一模型 SDK，而是通过 `TaskExecutor`（或等价统一调用层）执行子任务。
- 原因：统一模型配置读取、工具可用性、结果映射与错误处理，减少行为漂移。
- 备选方案：
  - 保留直连并增强：实现快但会继续形成第二套执行标准。
  - 新建独立 AgentRunner：扩展性强但当前投入过大。

### Decision 2: `skill` 工具从“信息展示”升级为“受控执行”

- 方案：读取技能定义后，先做调用约束校验（`user-invocable`、参数完整性），再构造执行指令并走统一执行链路。
- 原因：技能的价值在于复用可执行能力，不应停留在文本展示。
- 备选方案：
  - 继续仅返回技能文本：风险最低但不能满足能力目标。
  - 在工具层内完全自建执行器：会和主执行链路重复造轮子。

### Decision 3: 统一结果结构与错误分类

- 方案：`skill` / `agent` 都输出 `{ success, result, error, duration, metadata }` 风格数据，并通过现有 `mapToolResultToToolResultBlockParam` 转换。
- 原因：统一上游消费方式，便于 UI 展示、日志查询和测试断言。
- 备选方案：
  - 保持各自结构：迁移成本低，但长期维护和兼容成本高。

### Decision 4: 渐进式兼容，保持工具名称与核心入参稳定

- 方案：`skill` 名称与 `skill_name/args`，`agent` 名称与 `instruction/timeout` 保持兼容；新增字段通过可选参数扩展。
- 原因：避免影响已有 prompt/tool call 习惯。
- 备选方案：
  - 直接重构输入结构：更干净，但会引入兼容风险。

## Risks / Trade-offs

- [风险] 执行链路收敛后，`agent` 工具行为会与历史版本有差异（如错误文本、耗时统计）  
  → Mitigation: 保持字段兼容，新增回归测试覆盖典型失败场景。

- [风险] `skill` 执行语义增强后，潜在副作用操作变多  
  → Mitigation: 保留并强化调用约束校验，默认遵守既有工具权限边界。

- [风险] 复用统一执行栈可能放大其现有瓶颈  
  → Mitigation: 增加执行时长和错误类型埋点，后续按数据优化。

## Migration Plan

1. 先改造 `agent` 工具到统一执行链路，并保持输出字段兼容。
2. 改造 `skill` 工具执行流程与参数校验逻辑。
3. 补齐测试（工具接口合规 + 关键成功/失败路径）。
4. 更新文档并进行手工验证（至少覆盖一次 skill 调用和一次 agent 调用）。
5. 如线上行为异常，可回滚到上一版工具实现（保持接口兼容便于回退）。

## Open Questions

- `agent` 是否需要暴露“只读执行”或“禁止工具调用”模式作为可选开关？
