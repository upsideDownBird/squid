## Context

当前 jobopx-desktop 的工具系统存在以下问题：
- 工具输出直接返回完整内容，大文件读取或长命令输出会消耗大量上下文
- 缺乏统一的结果格式化机制，不同工具返回格式不一致
- 没有结果大小控制，可能导致 API 请求超限

claude-code-main 已经实现了成熟的工具结果管理系统：
- `toolResultStorage.ts` - 持久化大结果到磁盘，返回预览
- `Tool.mapToolResultToToolResultBlockParam` - 统一结果映射
- `enforceToolResultBudget` - 消息级别的结果预算控制

## Goals / Non-Goals

**Goals:**
- 实现工具结果持久化，大结果自动存储并返回预览
- 为所有工具添加统一的结果映射方法
- 实现消息级别的结果预算控制
- 建立工具开发规范文档

**Non-Goals:**
- 不改变工具的核心功能逻辑
- 不影响现有工具的输入参数
- 不实现 UI 层面的结果渲染（仅处理数据层）

## Decisions

### 1. 持久化存储位置
**决策**: 使用 `~/.squid/sessions/<sessionId>/tool-results/` 存储工具结果

**理由**:
- 按会话隔离，便于清理
- 与 claude-code-main 保持一致的目录结构
- 支持多会话并发运行

**替代方案**:
- 临时目录 `/tmp` - 可能被系统清理，不可靠
- 项目目录 - 污染工作区，不适合

### 2. 结果大小阈值
**决策**: 
- 默认单个工具结果阈值：50,000 字符
- 消息级别总结果阈值：150,000 字符
- 预览大小：2,000 字节

**理由**:
- 参考 claude-code-main 的经验值
- 50K 约等于 12,500 tokens，合理的单次工具输出
- 150K 消息级别预算防止并行工具调用时上下文爆炸

**替代方案**:
- 更小阈值（如 10K）- 过于激进，会频繁持久化小结果
- 更大阈值（如 100K）- 无法有效控制上下文

### 3. 映射方法设计
**决策**: 在 Tool 接口添加 `mapToolResultToToolResultBlockParam` 方法

```typescript
mapToolResultToToolResultBlockParam(
  content: Output,
  toolUseID: string
): ToolResultBlockParam
```

**理由**:
- 每个工具自己决定如何格式化输出
- 支持文本、JSON、错误等多种格式
- 与 Anthropic SDK 的 `ToolResultBlockParam` 类型对齐

### 4. 预算控制策略
**决策**: 实现两级预算控制
1. 工具级别：单个工具结果超过阈值时持久化
2. 消息级别：一条消息中所有工具结果总和超过阈值时，选择最大的结果持久化

**理由**:
- 工具级别处理单个大结果
- 消息级别处理并行工具调用场景
- 两级结合确保上下文始终可控

### 5. 向后兼容性
**决策**: 保持现有工具的 `call` 方法签名不变，新增映射方法

**理由**:
- 最小化破坏性变更
- 允许渐进式迁移
- 未实现映射方法的工具使用默认实现

## Risks / Trade-offs

### 风险 1: 磁盘空间占用
**风险**: 大量工具结果持久化可能占用大量磁盘空间

**缓解措施**:
- 按会话隔离，会话结束后可清理
- 实现定期清理机制（保留最近 N 天）
- 添加磁盘空间监控和告警

### 风险 2: 文件 I/O 性能
**风险**: 频繁的文件读写可能影响性能

**缓解措施**:
- 只有超过阈值的结果才持久化
- 使用异步 I/O，不阻塞工具执行
- 预览生成在内存中完成，不需要重新读取文件

### 风险 3: 迁移成本
**风险**: 所有现有工具需要实现新的映射方法

**缓解措施**:
- 提供默认实现，简单工具无需修改
- 提供工具开发指南和示例
- 分阶段迁移，优先处理高频工具

### Trade-off: 复杂性 vs 效率
**权衡**: 增加了系统复杂性（持久化、预算控制），但显著提升了上下文效率

**选择**: 接受复杂性，因为上下文效率是核心需求

## Migration Plan

### 阶段 1: 基础设施（第 1-2 天）
1. 实现 `ToolResultStorage` 类
2. 扩展 `Tool` 接口
3. 实现 `enforceToolResultBudget` 函数

### 阶段 2: 工具迁移（第 3-4 天）
1. 为核心工具实现映射方法：
   - ReadFileTool
   - GrepTool
   - GlobTool
2. 测试持久化和预算控制

### 阶段 3: 文档和规范（第 5 天）
1. 编写工具开发指南
2. 更新 CLAUDE.md
3. 添加示例代码

### 回滚策略
- 保持旧的 `call` 方法不变，可以快速回退
- 持久化功能可通过配置开关禁用
- 文件存储不影响核心功能，删除即可

## Open Questions

1. **清理策略**: 何时清理旧的持久化文件？
   - 建议：会话结束后保留 7 天，或总大小超过 1GB 时清理最旧的

2. **错误处理**: 持久化失败时如何处理？
   - 建议：降级到返回截断的结果，记录错误日志

3. **并发控制**: 多个工具同时写入时如何避免冲突？
   - 建议：使用 UUID 作为文件名，天然避免冲突
